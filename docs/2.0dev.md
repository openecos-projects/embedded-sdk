# ECOS 嵌入式 SDK (2.0 dev) - 工程结构与技术实现文档

本文档全面梳理了 `/home/timmo/Documents/ecos/embedded-sdk` 当前版本的工程结构、业务逻辑以及详细的技术实现路线，作为 ECOS 嵌入式系统 SDK 开发与维护的参考指南。

## 1. 工程结构 (Directory Structure)

当前 ECOS SDK 项目遵循模块化、分层的架构设计。主要目录结构如下：

- **`bin/`**: 包含命令行工具链（`ecos`, `ecos-init_project`, `ecos-set_board`），用于工程创建、配置和板级支持切换。
- **`board/`**: 板级支持包 (BSP) 目录。存放各个硬件开发板（如 StarrySkyC1, StarrySkyC2, StarrySkyL3）特定的启动代码 (`start.s`)、链接脚本 (`sections.lds`)、板级头文件 (`board.h`) 及特定 Makefile。
- **`components/`**: 核心系统组件与基础外设库。如 `gpio`, `hp_uart`, `i2c`, `pwm`, `qspi`, `timer`，以及基础 C 库 (`libc`) 和编译器数学运算库 (`libgcc`)。
- **`devices/`**: 外部设备驱动库。包含如传感器驱动（`gy_sgp30`）和屏幕显示驱动（`st7735`, `st7789`）。
- **`docs/`**: SDK 官方文档及 API 使用说明手册。
- **`example/`**: 官方提供的各类外设测试与示例应用（如 `hello_world`, `donut`, `minesweeper` 等），方便开发者快速上手。
- **`hal/`**: 硬件抽象层 (Hardware Abstraction Layer)。为底层硬件操作（如 `gpio`, `sys_uart`）提供统一的软件接口抽象，屏蔽底层硬件差异。
- **`templates/`**: 项目模板库。当使用 `ecos init_project` 命令时，会从这里拉取对应的代码框架和配置文件以生成新工程。
- **`tools/`**: 构建系统及工具集。包含基于 `kconfig` 的配置界面生成工具、`scripts` 目录下的 Makefile 构建规则，以及用于依赖处理的 `fixdep`。
- **根目录文件**: 包含安装配置脚本 `install.sh`，许可证 `LICENSE`，以及读我文档 `README.md` / `README_CN.md`。

## 2. 业务逻辑 (Business Logic)

ECOS 嵌入式 SDK 的核心业务逻辑是**提供一套从工程生成、硬件配置到固件编译烧录的完整、自动化 RISC-V 嵌入式开发工作流**。

### 2.1 环境初始化
通过根目录下的 `install.sh` 脚本，开发者能够一键配置 SDK 的环境变量，将 `bin/` 目录下的工具加入系统 `PATH`，从而激活 `ecos` 全局命令行工具。

### 2.2 工程创建与管理
1. **创建阶段**: 用户通过 `ecos init_project <template_name>` 创建项目。CLI 工具从 `templates/` 中检索对应模板，复制到用户指定的开发目录。
2. **硬件绑定**: 默认情况下或通过 `ecos set_board <board_name>`，系统将选定开发板（如 `StarrySkyC1`）在 `board/` 目录下的硬件特定配置（如链接脚本、启动汇编）部署或链接到当前工程。
3. **参数配置**: 提供 `make menuconfig` (底层基于 `tools/kconfig`)，支持以图形化菜单的方式裁剪组件和配置系统参数，最终生成 `.config` 文件和自动生成的宏定义头文件。

### 2.3 固件构建
项目使用统一的基于 Makefile 的构建系统 (`tools/scripts/build.mk` 等)。
- **编译器**：默认调用 `riscv32-unknown-elf-gcc` 工具链。
- **依赖解析**：通过 `tools/fixdep` 自动跟踪和生成 C/C++ 源文件的头文件依赖。
- **链接过程**：将 `components/` 基础库、`hal/` 抽象层、`devices/` 驱动层以及用户应用代码编译为对象文件，随后使用 `board/` 中提供的 `sections.lds` 链接成 `retrosoc_fw.elf`，并转换为 `.hex` 和 `.bin` 以供烧录。

## 3. 详细技术实现路线 (Technical Implementation Route)

### 3.1 架构分层设计
系统采用严格的分层架构，自底向上分为以下几层：
1. **硬件层 (Hardware)**: RISC-V rv32imac 核心 (PicoRV32)，运行于 72MHz。外设映射在指定的 MMIO 内存空间（基础外设基址 0x03000000 起）。
2. **板级支持层 (BSP)**: 位于 `board/`，提供最基础的 C 运行环境准备（设置栈指针，清零 BSS 段，复制 Data 段），通过 `start.S` 实现复位中断入口。
3. **硬件抽象层 (HAL) & 核心组件**: 位于 `hal/` 与 `components/`。将具体的寄存器操作 (`reg_xxx_ctrl`) 封装为如 `gpio_init()`, `hp_uart_send()` 等通用 API 函数，隔离了直接的物理地址访问。
4. **设备驱动层 (Device Drivers)**: 位于 `devices/`。依赖 HAL/Components 层与外设通信（如通过 SPI 协议驱动 ST7735 屏幕），向上提供设备级的控制接口。
5. **应用层 (Application)**: 位于用户的 `main.c`，调用各类 API 实现具体的业务逻辑（如扫雷游戏 `minesweeper`）。

### 3.2 内存布局与链接技术
代码在 Flash 中执行 (XIP) 或加载至 RAM：
- **Flash (0x30000000)**: 存储 `.text` (代码) 和 `.rodata` (只读数据)。
- **SRAM/PSRAM (0x04000000)**: 存储 `.data` (已初始化全局变量) 和 `.bss` (未初始化全局变量)。
启动时，`start.s` 负责根据链接脚本生成的符号边界 (`_sdata`, `_edata`, `_sbss`, `_ebss`)，将 Flash 中的数据段搬运到 RAM 中，并初始化堆栈，然后跳转至 `main` 函数执行。

### 3.3 构建工具链集成
- **Kconfig 系统**: 移植自 Linux 内核的 Kconfig 系统，使得 SDK 具备极强的可裁剪性。`conf` 工具解析 `Kconfig` 文件，生成 C 语言的配置头文件。
- **Makefile 框架**: 利用 `$(CC) -MMD` 和 `fixdep` 工具实现精确的增量编译。所有的编译规则被抽象在 `tools/scripts/build.mk` 中，用户工程的 Makefile 仅需定义 `SRCS` (源文件列表) 和 `INC_PATH` (头文件路径)，极大降低了开发者的构建维护成本。

## 4. 演进方向 (2.0 dev 展望)
在当前的底层设施之上，未来的技术演进将聚焦于：
1. **HAL 层的全面完善**: 将更多组件（如 I2C, SPI）完全纳入统一的 HAL 接口标准规范。
2. **RTOS 集成**: 提供 FreeRTOS 或 RT-Thread 的可选支持，引入多任务调度能力。
3. **更多的外设生态**: 扩充 `devices/` 目录，增加更多传感器和执行器的开箱即用驱动支持。
